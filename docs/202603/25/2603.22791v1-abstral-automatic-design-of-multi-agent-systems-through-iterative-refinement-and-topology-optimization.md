## 摘要
多智能体系统应如何设计？这种设计知识能否以一种可检查、可修订且可迁移的形式被捕获？我们介绍了 ABSTRAL，这是一个将多智能体系统（MAS）架构视为演进式自然语言文档的框架，该文档是通过对比轨迹分析（contrastive trace analysis）进行细化的产物。研究得出了三项发现：首先，我们对多智能体协调成本（coordination tax）进行了精确测量：在固定轮数预算下，集成系统仅达到 26% 的轮数效率，66% 的任务耗尽了限制，但通过发现可并行化的任务分解，其表现仍优于单智能体基准。其次，编码在文档中的设计知识具有可迁移性：在一个领域学习到的拓扑推理和角色模板为新领域提供了先发优势，迁移后的种子在单次迭代中即可达到冷启动第 3 次迭代的性能。第三，对比轨迹分析能够发现初始设计中不存在的专家角色，这是以往系统均未展示的能力。在 SOPBench（包含 134 个银行任务，具有确定性真值）上，以 GPT-4o 为骨干模型的 ABSTRAL 达到了 70% 的验证集通过率和 65.96% 的测试集通过率。我们发布了收敛后的文档，作为可检查的设计原理。

## Abstract
How should multi-agent systems be designed, and can that design knowledge be captured in a form that is inspectable, revisable, and transferable? We introduce ABSTRAL, a framework that treats MAS architecture as an evolving natural-language document, an artifact refined through contrastive trace analysis. Three findings emerge. First, we provide a precise measurement of the multi-agent coordination tax: under fixed turn budgets, ensembles achieve only 26% turn efficiency, with 66% of tasks exhausting the limit, yet still improve over single-agent baselines by discovering parallelizable task decompositions. Second, design knowledge encoded in documents transfers: topology reasoning and role templates learned on one domain provide a head start on new domains, with transferred seeds matching coldstart iteration 3 performance in a single iteration. Third, contrastive trace analysis discovers specialist roles absent from any initial design, a capability no prior system demonstrates. On SOPBench (134 bank tasks, deterministic oracle), ABSTRAL reaches 70% validation / 65.96% test pass rate with a GPT-4o backbone. We release the converged documents as inspectable design rationale.

目前多智能体系统（MAS）的设计主要依赖人工经验。现有的自动化方法（如 ADAS、AFlow）通常将中间知识存储为代码、提示词或神经权重，这些形式难以被人类检查、修订或在不同领域间迁移。

### 2. 方法论：核心思想与技术细节
ABSTRAL 提出了一个三层架构的连续设计循环，其核心载体是结构化的 Markdown 文档：

*   **核心载体 (`SKILL.md`)**：包含四个部分：

    *   **R (拓扑推理)**：条件规则（如“若任务具有属性 P，则倾向于拓扑 T”）。
    *   **T (角色模板)**：发现的专家角色及其系统提示词、工具权限。
    *   **P (构建协议)**：构建可运行智能体图的步骤。
*   **三层循环机制**：
    1.  **内层循环（轨迹驱动细化）**：
        *   **构建 (Build)**：元智能体根据文档生成智能体规范。
        *   **运行 (Run)**：在沙盒中执行任务，捕获全量执行轨迹。
        *   **分析 (Analyze)**：对比失败与成功的轨迹，将失败归类为 5 种证据类别（EC1-EC5，如 EC3 代表需要新专家角色）。
        *   **更新 (Update)**：根据证据对文档进行针对性编辑。
    2.  **中层循环（收敛与整合）**：监测性能平台期或规则冗余，执行“整合（Consolidation）”操作，合并冗余规则，防止语义漂移。

### 6. 主要结论与发现
1.  **协调税（Coordination Tax）的量化**：首次精确测量了多智能体开销。在固定轮数预算下，多智能体系统仅有 **26% 的轮数效率**（1/4 的轮数在做实际工具调用），66% 的任务因达到轮数限制而失败。

4.  **性能提升**：在 GPT-4o 骨干下，ABSTRAL 达到了 **65.96% 的测试通过率**，优于原始 GPT-4o 单智能体基准（58.96%）。

### 7. 优点
*   **可解释性与透明度**：将 MAS 设计转化为 Markdown 文档，使得设计决策（如为什么选择某种拓扑）对人类透明且可干预。
*   **拓扑多样性保证**：引入 GED 排斥机制，有效解决了自动化设计中容易收敛到初始拓扑的问题。
*   **实用的效率洞察**：对“协调税”的量化提醒开发者，多智能体系统仅在任务可并行分解时才具有净收益。

*   **轮数预算敏感性**：由于多智能体通信占用了大量轮数，在严格限制轮数的场景下，MAS 可能反而不如单智能体。

*   **样本方差**：验证集规模较小（20-40 任务），导致性能曲线存在波动。
